# SamQL audit — build 2026-06-24.148

Scope: UI, performance, memory, the node set, and chart types — with a focus on
charts for **day-over-day** (period-over-period) analysis. Findings are grounded
in the current source; file/line references are given so each item is checkable.

Two honesty notes up front. (1) This sandbox can't run DuckDB, Vite, a browser,
or ECharts, so everything below is from static reading + the test harness, not a
live profile. Anything labelled a "performance" or "memory" item is an
*optimization opportunity*, not an observed regression. (2) Items are sorted
roughly by value-to-effort, not severity — the app is in good shape overall.

---

## 0. Headline (the day-over-day ask)

**Computation already exists; the *visual* is the gap.** The `window` node does
`lag`/`lead` (nodeflow.py `_WINDOW_FUNCS`, lines 53-54) with partition + order-by
in its inspector, and there's a `diff` op — so "today minus yesterday" is already
a two-node recipe (window `lag` ordered by date → formula `value - value_lag`).

What's missing is a chart designed to *show* a change. Today's 12 types are all
level/where-it-stands charts. None emphasises the delta. **Recommendation: add a
"Change (Δ) bar" chart type** (and optionally a waterfall). Both can be built
entirely in `chartOption.ts` (pure, already unit-tested) from the labels+values
the backend already returns — **no backend change, no new dependency**. Details
in §4.

## 1. Charts

### What's solid
- 12 types: bar, line, area, pie, donut, scatter, treemap, tree, candlestick,
  histogram, multi-x, multi-y. All ECharts modules are now registered (.148) and
  guarded by `t_chart_modules_registered`, so the "blank tree" class can't recur.
- `buildChartOption` is pure and covered by 24 assertions; theming (light/dark),
  palettes, per-series colours, gradients, rounded/stacked/smooth variants.
- Point counts are capped: scatter at 5000 (session.py `_chart_sql`), grouped
  series at 100000 (`agg_group_multi(... cap=100000)`). Large line/area get LTTB
  downsampling. So charts won't choke the renderer on big inputs.

### Opportunities
1. **Charts don't follow the app theme.** There's an app-wide light/dark switch
   (App.tsx ~1987, `ivoryCanvas`/`ivoryEditor`), but `style.theme` on a chart is
   per-chart and defaults to dark. In light mode, a freshly added chart is dark
   until you flip its own theme. Low effort: default a chart's theme to the app
   theme (read the same flag) unless the user overrode it.
2. **No day-over-day / delta visual** — see §0 and §4.
3. **Minor:** the tree chart's root label is blanked when a title is set
   (chartOption.ts ~213). Harmless, but the root then shows as an unlabeled dot;
   consider keeping the x-name even with a title.

## 2. Nodes

### What's solid
~47 nodes covering ingest (input/inputs/directory/appendfolder), shape
(select/rename/fill/coalesce/split/parse/explode/jsonextract/maprecode/textclean),
combine (join/antijoin/crossjoin/multijoin/union), aggregate (group/summarize/
pivot/unpivot/groupconcat/bin), analytic (window/rank/topn/sample/dedupe/unique/
sort), and QA (validate/reconcile/profile). That's a genuinely complete analytics
palette; few tools this size cover unpivot + reconcile + profile.

### Gaps worth considering
1. **A first-class "period delta" node.** Day-over-day is common enough that a
   dedicated node (pick a value, a date/order column, optional partition, and a
   mode: absolute Δ, % change, or running total) would beat the window+formula
   recipe for discoverability. It's just a thin wrapper over the SQL the window
   node already emits.
2. **As-of / nearest join.** Time-series work often needs "join each row to the
   most recent prior row in B" (`ASOF JOIN` in DuckDB). Not expressible with the
   current equi-joins. Medium effort, DuckDB-only (SQLite would need a fallback
   or a "DuckDB required" notice).
3. **Projection pushdown skips groups.** `_node_needed_inputs` returns `None` for
   `group` (nodeflow.py ~1470), so column-pruning stops at a group boundary. Not
   wrong — just leaves some perf on the table for wide inputs feeding a group.

## 3. Performance

### What's solid
- Flow cache: fingerprint-keyed, LRU, with fusion + projection pushdown and
  forced checkpoints for blocking joins (`_materialize_flows`). A data mutation
  bumps an epoch and clears it (`_invalidate_counts`).
- Query interruption: every run path registers a `query_id` and catches the
  engine interrupt, so cancel doesn't surface as an error (recent .145 fix).
- Result grid is **virtualized** (DataGrid.tsx: `visible` window via `scrollTop`/
  `viewH`, full-height spacer, absolute row positioning) — 5000-row pages render
  only the visible slice. This is the right design; no change needed.

### Opportunities
1. **NodeFlow re-renders.** The canvas component is ~9,250 lines with only ~10
   `useMemo`/`useCallback`. `WireLayer` is memoized (good), but most handlers are
   re-created each render. On a large graph this can mean re-rendering every node
   on each drag tick. Worth profiling first; if it bites, memoize the per-node
   render and the handlers passed into it, and consider splitting the node body
   into a `React.memo` child keyed by node id + its slice of state.
2. **Group projection pushdown** (see §2.3) — prune columns through groups.
3. **Chart fetch granularity** is fine, but a "max distinct categories" guard on
   bar/pie (today only the 100k row cap applies) would prevent a 50k-category bar
   chart from being technically valid but useless + slow to draw.

## 4. Day-over-day charts — concrete proposal

Recommended, in value order:

1. **Change (Δ) bar** — a new `chart_type: "delta"`. Takes one value series
   (e.g. daily totals you already produce with a group/summarize) and plots the
   consecutive difference `v[i] - v[i-1]` as signed bars: green above zero, red
   below, zero line emphasised. First bar is 0 (or omitted). This is the single
   clearest "what changed each day" visual and is **pure-frontend** — built in
   `chartOption.ts` from existing `labels`+`values`, testable in the existing
   harness, no backend or dependency change. ~30 lines + tests.
2. **% change toggle** on the delta chart — plot `(v[i]-v[i-1]) / v[i-1]` instead
   of the absolute difference, with a `%` axis. A `style` flag; trivial once #1
   exists.
3. **Waterfall** — `chart_type: "waterfall"`. ECharts recipe: a transparent
   "base" series carrying the running total + a visible "delta" series stacked on
   top, coloured by sign. Shows how each day's change builds the cumulative
   total. Also pure-frontend from labels+values. ~40 lines + tests.
4. **Bars + change line** is already possible via **multi-y** today (value as
   bars on the left axis, the lag-diff column as a line on the right). Worth
   documenting as the no-new-code option for users who want both at once.

All four lean on the window node for the underlying numbers when needed, and #1/#3
need nothing new on the backend.

## 5. Memory

### What's solid
- Temp-file lifecycle was audited end-to-end (.120): row stores, parquet temps,
  export files, JSON-flatten spill dirs, instance dirs; `sweep_stale()` runs on
  graceful shutdown and `clear_all(clear_manifest=True)` drops tables + clears
  the restore manifest on clean exit (.119). Crash recovery is preserved because
  the manifest only survives an abnormal exit.
- Grid virtualization (above) keeps DOM/node memory flat regardless of page size.

### Opportunities
1. **Flow cache eviction is count-based only.** `_flow_cache_put` evicts by count
   (`flow_cache_max`, session.py ~496) — N cached *tables* regardless of their
   size. A handful of wide/long cached results can use a lot of disk/RAM under a
   small N. Consider a size-aware cap (track approx bytes/rows per entry and evict
   to a byte budget) and/or a TTL.
2. **Expose the cache cap in Settings.** It's an internal constant today; surfacing
   it (plus the existing hit/miss/evict stats from `flow_cache_info`) gives users
   a memory dial.

## 6. UI / UX

### What's solid
Light/dark toggle, in-canvas delete confirm (.145), edge-delete-on-hover (.144),
dynamic group ports (.148), centered/enlarged ports with own-colour hover (.147),
group multi-input with per-step routing (.146). The node canvas is feature-rich.

### Opportunities (low effort, high polish)
1. **Per-column grid controls** — click-to-sort and a per-column filter on the
   result grid. The grid is already virtualized, so this is mostly UI.
2. **Command palette (Ctrl+K)** — fuzzy "add node / jump to node / run" for a
   ~47-node palette; big discoverability win.
3. **Charts follow app theme** (§1.1).
4. **Keyboard nav on the canvas** — arrow-nudge selected nodes, Enter to edit,
   already have Delete. Cheap accessibility gain.
5. **A first-class period-delta node** (§2.1) — discoverability for the most
   common time analysis.
